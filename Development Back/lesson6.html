What is HATEOAS?

One of the primary reasons to use REST is the ability to navigate the entire set of resources without prior knowledge of the URI schema. This principle is known as HATEOAS, or Hypertext as the Engine of Application State.

The system is effectively a finite state machine, and the response to each request contains the information necessary to move from one state to another.

Key Principle: Each HTTP GET request should return the information necessary to find the resources related directly to the requested object through hyperlinks included in the response.

Core Concept

  • Client doesn't need to know URI structure in advance
  • Server provides available actions via hyperlinks
  • Responses contain navigation information
  • API becomes self-documenting and discoverable

Why Use HATEOAS?

HATEOAS enables clients to navigate an API dynamically, without hard-coding URIs or understanding the entire API structure upfront.

Benefits

  • Decoupling: Clients don't need to know URI patterns
  • Evolvability: Server can change URIs without breaking clients
  • Discoverability: Clients discover available actions from responses
  • Self-documentation: API responses show what's possible
  • Reduced coupling: Changes to URI structure don't affect clients
Note: There are no general-purpose standards that define how to model the HATEOAS principle. The examples shown are one possible, proprietary solution.

HATEOAS JSON Example

For example, to handle the relationship between an order and a customer, the representation of an order might include links that identify the available operations:

{ "orderID": 3, "productID": 2, "quantity": 4, "orderValue": 16.60, "links": [ { "rel": "customer", "href": "https://api.contoso.com/customers/3", "action": "GET", "types": ["text/xml", "application/json"] }, { "rel": "customer", "href": "https://api.contoso.com/customers/3", "action": "PUT", "types": ["application/x-www-form-urlencoded"] }, { "rel": "customer", "href": "https://api.contoso.com/customers/3", "action": "DELETE", "types": [] }, { "rel": "self", "href": "https://api.contoso.com/orders/3", "action": "GET", "types": ["text/xml", "application/json"] } ] }

The links array contains a set of links. Each link represents an operation on a related entity.

Link Structure Explained

Each link in the links array contains specific information that the client application needs to invoke the operation.

Link Components

  • rel - The relationship ("customer", "self", "next", "prev")
  • href - The URI to access the resource
  • action - The HTTP method (GET, POST, PUT, DELETE)
  • types - Supported MIME types for the operation

Common Relationship Types

RelationMeaning
selfLink to the current resource
nextLink to the next page
prevLink to the previous page
customerLink to related customer resource
itemsLink to collection of items

Application State Machine

The set of links that are returned can change depending on the state of the resource. The idea that hypertext is the engine of application state describes this scenario.

Dynamic Links Based on State

For example, an order might have different available actions depending on its status:

// Order status: "pending" { "orderID": 3, "status": "pending", "links": [ {"rel": "cancel", "href": "/orders/3", "action": "DELETE"}, {"rel": "update", "href": "/orders/3", "action": "PUT"}, {"rel": "submit", "href": "/orders/3/submit", "action": "POST"} ] } // Order status: "shipped" { "orderID": 3, "status": "shipped", "links": [ {"rel": "track", "href": "/orders/3/tracking", "action": "GET"}, {"rel": "return", "href": "/orders/3/return", "action": "POST"} ] // Note: No cancel or update links - order can't be changed }

Workflow Control

  • Server controls available transitions
  • Client discovers actions from response
  • Links change based on business rules
  • Prevents invalid state transitions
Best Practice: Include only the links that are valid for the current state of the resource. This prevents clients from attempting invalid operations.
